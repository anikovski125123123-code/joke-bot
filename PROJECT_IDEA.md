# Joke Bot — Project Idea Document

## 1. Проблема

Друзья часто испытывают стресс, скуку или эмоциональное выгорание в течение рабочего дня.
Им не хватает простых, дозированных способов «перезагрузки» мозга, которые **не требуют долгого скроллинга** ленты социальных сетей (где легко застрять надолго).

### Симптомы проблемы
- Накопленная усталость к середине дня (13:00–15:00)
- Ощущение монотонности при рутинных задачах
- Желание отвлечься, но нежелание «провалиться» в соцсети на 30+ минут

---

## 2. Решение

**Telegram-бот**, который в заданное время присылает в общий чат качественный, отобранный анекдот.

### Ценностное предложение
- Психологическая разрядка за **30 секунд**
- Минимальное время на взаимодействие — бот сам приходит к тебе
- Отсутствие визуального шума (нет лент, рекламы, рекомендаций)
- Целевая доставка юмора в момент, когда он наиболее нужен

### Почему Telegram, а не другое?
- Друзья уже общаются в Telegram
- Групповой чат — не новый канал коммуникации, а уже привычная среда
- Боты в Telegram не требуют установки дополнительных приложений

---

## 3. Целевая аудитория

| Пользователь | Роль | Ожидание |
|---|---|---|
| Артём | Участник чата | Получает анекдот в нужное время |
| Петя | Участник чата | Получает анекдот в нужное время |
| Администратор (я) | Владелец бота | Управляет расписанием и контентом |

**Размер аудитории:** 2–5 человек. Масштабирование не планируется на начальном этапе.

---

## 4. План запуска — три фазы

---

### Фаза 1: MVP — Бот с расписанием

**Цель:** запустить работающий бот с минимальным функционалом.

**Функциональность:**
- Бот добавляется в групповой чат
- В заданное время (например, 13:00 каждый день) отправляет анекдот из заготовленного пула
- Анекдоты хранятся в JSON-файле, выбираются по порядку (без повторов)
- Команда `/start` для проверки, что бот живой

**Критерии готовности:**
- [ ] Бот отправляет анекдот точно в назначенное время
- [ ] Бот не падает при перезапуске
- [ ] Логируется факт каждой отправки

**Примерное время разработки:** 1–2 дня

---

### Фаза 2: Админка — управление расписанием

**Цель:** дать администратору возможность управлять расписанием без изменения кода.

**Функциональность (команды доступны только администратору):**

| Команда | Описание |
|---|---|
| `/schedule add HH:MM <текст>` | Добавить новую задачу на отправку |
| `/schedule list` | Показать все активные задания |
| `/schedule delete <id>` | Удалить задание по ID |
| `/schedule pause <id>` | Приостановить задание |
| `/schedule resume <id>` | Возобновить задание |
| `/log` | Показать последние 10 отправленных сообщений |

**Хранилище данных (SQLite):**
```
Таблица: schedules
- id INTEGER PRIMARY KEY
- time TEXT (HH:MM)
- message TEXT
- is_active BOOLEAN
- created_at DATETIME

Таблица: send_log
- id INTEGER PRIMARY KEY
- schedule_id INTEGER
- sent_at DATETIME
- chat_id TEXT
- message TEXT
```

**Критерии готовности:**
- [ ] Администратор может добавить расписание через чат с ботом
- [ ] Расписание сохраняется после перезапуска бота
- [ ] Дата, время и текст строго соблюдаются

**Примерное время разработки:** 2–3 дня

---

### Фаза 3: AI-шутки по запросу

**Цель:** дать пользователям возможность запросить новый анекдот в любой момент.

**Функциональность:**
- В сообщении бота появляется инлайн-кнопка **«Нужна шутка»**
- При нажатии бот обращается к OpenAI API и генерирует свежий анекдот
- Анекдот отправляется в чат от имени бота
- **Cooldown:** кнопка работает не чаще 1 раза в 5 минут (защита от спама)
- Если OpenAI недоступен — бот берёт анекдот из статического пула (fallback)

**Критерии готовности:**
- [ ] Кнопка появляется под каждым сообщением бота
- [ ] Сгенерированный анекдот релевантен и безопасен
- [ ] Cooldown работает корректно

**Примерное время разработки:** 2–3 дня

---

## 5. Риски и митигация

| Риск | Вероятность | Влияние | Решение |
|---|---|---|---|
| Бот не отправляет шутки вовремя | Средняя | Высокое | APScheduler с `misfire_grace_time`; лог каждой задачи; мониторинг |
| Потеря расписания при рестарте | Высокая | Высокое | Хранение в SQLite, восстановление scheduler при старте |
| Анекдоты повторяются | Средняя | Низкое | Циклический перебор пула с отметкой использованных |
| OpenAI API недоступен | Низкая | Среднее | Fallback на статический пул, пользователь не замечает |
| Бот упал и никто не знает | Низкая | Высокое | Watchdog / автоперезапуск (systemd / Railway) |

---

## 6. Технические ограничения и решения

- **Часовой пояс:** расписание хранится в UTC, отображается в UTC+3 (МСК) для удобства администратора
- **Деплой:** бот должен работать 24/7 — Railway.app (бесплатный tier) или VPS с systemd
- **Масштаб:** до 10 пользователей, нет необходимости в Redis или очередях

---

## 7. Метрики успеха

- Бот работает без падений 7 дней подряд
- Анекдот отправляется не позже чем через 30 секунд после назначенного времени
- Артём и Петя не жалуются на повторы в течение первых 30 дней

---

## 8. Out of Scope (не делаем)

- Персональные уведомления каждому пользователю отдельно
- Регистрация / профили пользователей
- Рейтинги анекдотов / лайки
- Веб-интерфейс для администратора
